# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A collection of standalone Python scripts implementing the image-analysis pipeline for the "ME" paper (authors: Max Riekeles and Berke Santos, TU Berlin — riekeles@tu-berlin.de). The pipeline detects and tracks motile bacteria in time-series microscopy recordings, aligns detections to motion tracks, and extracts per-track kinematic parameters (velocity, straightness, direction changes, etc.).

There is no package, no build, no test suite, and no CLI. Each `.py` file is a self-contained batch script run directly with `python <script>.py`. **All configuration lives in hard-coded constants inside the `if __name__ == "__main__":` block (or module-level constants near the top) — there are no command-line arguments.** To run anything you edit the `root_directory` (and related paths/parameters) at the bottom of the file, then execute it.

Environment note: paths in the scripts are Linux (`/media/general-max-riekeles/MMT_3/ME/...`), but the working machine is Windows. Expect to rewrite `root_directory` for the local OS before running.

## Running

```bash
# each script is run directly; edit its hard-coded root_directory at the bottom first
python blob_detection/blob_detection_v3_memory_optimized.py
```

Dependencies are in `requirements.txt` (`pip install -r requirements.txt`): `opencv-python` (imported as `cv2`/`cv`), `numpy`, `scipy`, `scikit-image` (`skimage`), `scikit-learn` (`sklearn`), `matplotlib`, `Pillow` (`PIL`), `psutil`, `python-dateutil`. `scikit-learn` is used only for DBSCAN in `segmentation_corrections/correction_v2` and `segmentation_checks/error_analysis_v3`.

## Versioned scripts

Many scripts exist as `_v1`/`_v2`/`_v3`/`_v4`/`_v5` (and `_part1`/`_complete`) variants. **The highest version number is the current one**; lower versions are kept for history. When editing behavior, edit the latest version unless explicitly told otherwise.

Superseded scripts have been moved to `archive/`. The current blob detector is **`blob_detection_v3_memory_optimized.py`** (batch processing, `gc.collect()` between batches, per-experiment error tracking + `processing_summary`, psutil memory monitoring). Note its traversal walks the **`segmentation_output/` tree** and only processes recordings that already have a segmentation folder — recordings not yet in `segmentation_output` are silently skipped. The archived `_ME_complete`/`_v2` variants are the same 6-column detector but simpler (no batching/error-tracking) and walk the **data** tree instead; the archived `_part1` is the obsolete first generation that emits only 4 columns (`x,y,t,size`) with no brightness/solidity and is incompatible with the rest of the pipeline.

## The pipeline (data flow)

Scripts share one convention: a **`root_directory`** containing both the raw data folders and a parallel **`segmentation_output/`** tree that mirrors the data hierarchy. The hierarchy is always three levels:

```
subfolder (experiment, e.g. Lead_1, Copper_3, Nickel_1)
  └── shortened_<N>            (time-series experiment / interval)
       └── subsubfolder        (individual recording session, often suffixed "_changed")
```

Raw input lives under `<subfolder>/<shortened>/<subsubfolder>/raw/*.tif`, `validate/` (holds `MHI.npy` and coloured MHIs), etc. Processing artifacts are written into the mirrored `segmentation_output/<subfolder>/<shortened>/<subsubfolder>/`.

Approximate order of the stages:

1. **Track segmentation** — `segmentation/track_segmentation_v4.py`. Coloured MHIs (Motion History Images, which encode per pixel the time a bacterium moved through it) are produced upstream by an external tool — there is **no MHI-generation script in this repo**. `track_segmentation_v4` batch-walks the data root, finds each `validate/` folder's coloured MHI (`*mhi*_color.tif` / `*mhi*_colors.tif`), splits it into per-track PNGs (`t<N>.png`) by exact BGR-colour matching against a fixed ~67-colour time-gradient list, applies a `min_area` noise filter, and mirrors the results into `segmentation_output/<exp>/<shortened>/<rec>/`. Batch QA of the segmentation output lives in `segmentation_checks/`, and manual per-image fix-ups for mis-segmented tracks live in `segmentation_corrections/` (both described below).

   *Archived predecessors of this step, oldest to newest:* `track_segmentation_ancient_v1.py` (was `colours_MHIs.py` — 29-colour palette, **no `min_area` filter** so it keeps single-pixel noise, writes `analysed_tracks/<name>/track_<N>.png`) → `track_segmentation_ancient_v2.py` (was `colours_MHIs_validate_output.py` — same 29 colours / no filter, but pipeline-style `tracks/t<N>.png` naming, skips `median_image`) → `track_segmentation_v2.py` (67 colours + `min_area`, still manual single-folder) → `track_segmentation_v3.py` (batch, but `*_color.tif` only, one MHI per folder) → **`_v4`** (adds `*_colors.tif` + multiple-MHI-per-folder handling).
2. **PNG→NPY conversion** — `segmentation/npy_conversion_v1.py` walks `segmentation_output/…/<rec>/` and converts each recording's per-track segment PNGs (`t<N>.png`) into `npy_tracks/*.npy` (2048², 8-bit grayscale track masks) — the track segments the alignment stage reads. `MHI.npy` similarly represents the MHI.
3. **Blob detection** — `blob_detection/blob_detection_v3_memory_optimized.py`. Runs OpenCV `SimpleBlobDetector` (dark blobs on CLAHE image, bright blobs on inverted CLAHE) over every `raw/*.tif`. Writes one `blobs_<subsubfolder>.npy` per recording. **Tune detection with `blob_detection/blob_parameters_check_v3.py`** — a single-image, on-screen-only tool (no NPY output) whose CLAHE/threshold/area/`min_distance` settings mirror the production detector, so the values you find here transfer directly into the detector's `default_params`/`subfolder_params`. Its left panel = distance-filtered detection (what the detector saves); its right panel previews a *spatial* proximity + darkness/size/solidity selection (an exploratory approximation of the alignment stage's *temporal* filtering — not a literal production step). The archived `_v1`/`_v2` are superseded (v1 predates the 2048² resize and doesn't match the pipeline; v2 is v3 without the right-hand comparison panel).
4. **MHI–track–blob alignment** — `alignment_2/blob_mhi_tracks_alignment_v5.py`. For each track mask, keeps blobs that fall spatially on the track AND within `MHI_TIME_TOLERANCE` of the MHI time at that pixel; **replaces each blob's timepoint with the MHI time (temporal correction)**; then applies temporal filtering to keep exactly one blob per timepoint per track (darkest wins; ties broken by area+solidity). Output: `alignment_results_2/aligned_blobs_<track>.npy`. See the **Two alignment branches** section below for the v4/v5 fork.
5. **Frame rate** — `frame_rate/frame_rate_v1.py` parses the data folders' `*.xml` acquisition-metadata (first/last timestamps, sorted by `b0t<N>` index) to compute acquisition frequency, writing `average_frequency.txt` into each `segmentation_output/…/<rec>/`. **Shared prerequisite:** both branches' conversion step reads this file (frames→seconds), so it lives in its own `frame_rate/` folder rather than inside either `alignment_*` branch.
6. **Unit conversion** — `alignment_2/time_coordinates_conversions_v3.py` converts pixel coords → micrometers and frame numbers → seconds (using `average_frequency.txt`), accounting for the original 2560×1920 sensor, the 2048×2048 processing size, and the 40× objective blob size. Output: `alignment_converted_results_2/*_converted.npy`.
7. **Kinematic parameter extraction** — `kinematics/` (see **The `kinematics/` folder** below). **Unlike every other stage, this one is not configured by editing the script: `python extract_track_metrics.py` prompts for the root directory, the branch, and each analysis parameter** (or takes root+branch as command-line arguments), because root and branch are machine-specific and must not be hard-coded into the repo. The analysis parameters are prompted too, each pre-filled from its `kinematics/kin_config.py` default (Enter keeps it); those `kin_config.py` constants remain the defaults offered by the prompt and the values used by unattended runs. One invocation runs the whole stage. It reads `alignment_converted_results[_2]/` and writes per-track CSVs to `<root>/processed_results[_2]/`, then calls `summarize_track_metrics.py`, which reads those CSVs and writes the grouped summary statistics next to them. Tracks are grouped into experimental conditions by parsing subsubfolder names (`species_solute_time_sampleX_recY_changed`). This replaces the two per-branch scripts `alignment_1/final_kin_param_extraction_v3.py` and `alignment_2/final_kin_param_extraction_v4.py`, which remain in place (unmodified, and not to be edited) until the new stages are validated on the full dataset, after which they move to `archive/`.

Utilities: `blob_detection/check_blobs_npy.py` (count blobs at a given timepoint in one `blobs_*.npy`), `blob_detection/blob_detection_cleanup.py` (⚠️ destructive — recursively `rmtree`s any `segmentation_output/…` subsubfolder that is empty or holds only `blobs_*.npy`, so the detector reprocesses it; previews then auto-proceeds with **no confirmation**), `utils/copy_rename_MHIs.py` (copies `.tif`s to `{base}_tracks.tif` in a `coloured_MHIs/` subfolder — input-prep before segmentation) and `utils/copy_color_mhis_and_normal_mhis.py` (collects each `validate/` folder's `*_color.tif`, or `*_mhi.png` fallback, into a destination dir).

### Two alignment branches — `alignment_1/` and `alignment_2/`

Stages 4 (alignment) and 6 (unit conversion) each exist in two intentionally-kept variants that form **two parallel end-to-end branches**. The alignment step and its matching conversion step are grouped together per branch:

| | `alignment_1/` — **uncorrected** | `alignment_2/` — **corrected (canonical)** |
|---|---|---|
| Alignment | `..._blob_mhi_tracks_alignment_v4.py` | `..._blob_mhi_tracks_alignment_v5.py` |
| Overlay (viz) | `..._mhi_overlay_v2.py` | `..._mhi_overlay_v3.py` |
| Conversion | `..._time_coordinates_conversions_v2.py` | `..._time_coordinates_conversions_v3.py` |
| Overlay collect | `..._mhi_overlay_copies_v1.py` | `..._mhi_overlay_copies_v2.py` |
| Kinematics (stage 7) | *superseded by* `kinematics/` (`BRANCH = "uncorrected"`); old: `..._final_kin_param_extraction_v3.py` | *superseded by* `kinematics/` (`BRANCH = "corrected"`); old: `..._final_kin_param_extraction_v4.py` |
| Blob timepoints | **preserved** (original detection time) | **replaced with MHI movement-time** (fixes high-velocity artifacts) |
| Alignment reads/writes | reads `blobs_*.npy` → `alignment_results/` (+ `alignment_summary.txt`) | reads `blobs_*.npy` → `alignment_results_2/` (+ `alignment_summary_corrected.txt`) |
| Overlay reads/writes | `alignment_results/` → `mhi_overlay.png` | `alignment_results_2/` → `mhi_overlay_corrected.png` |
| Conversion reads/writes | `alignment_results/` → `alignment_converted_results/` | `alignment_results_2/` → `alignment_converted_results_2/` |
| Overlay collect reads/writes | `mhi_overlay.png` → `<root>/mhi_overlays/` | `mhi_overlay_corrected.png` → `<root>/mhi_overlays_corrected/` |
| Kinematics reads/writes | `alignment_converted_results/` → `<root>/processed_results/` | `alignment_converted_results_2/` → `<root>/processed_results_2/` |

Within each branch the five scripts (align → overlay → convert → collect → kinematics) are near-identical to the other branch's except for which `..._results[_2]/` folders and `mhi_overlay[_corrected].png` files they read and write; the actual logic is the same. Two exceptions where the branch scripts genuinely diverge in logic: the archived overlay v1 (missing 2048² resize) and the archived kinematics v2 (weaker `extract_sample_group` regex — v3/v4 also match `sampleX_..._recY` names with descriptors between, falling back to v2's exact pattern). The "overlay collect" and "kinematics" steps write to root-level folders (`mhi_overlays[_corrected]/`, `processed_results[_2]/`), gathering across all experiments — not per-branch data. **The corrected branch (`alignment_2/`) is canonical** — its `alignment_converted_results_2/` output is what the kinematic extraction (stage 7) consumes. `alignment_1/` exists only to reproduce the uncorrected-timing result for comparison; run it end-to-end (v4 → v2 overlay → v2 convert) if you want that, otherwise use `alignment_2/`.

> **Overlay note:** the overlay scripts (`..._mhi_overlay_v2/v3`, renamed from the original misleading `..._mhi_alignment_*`) do **no alignment** — they draw the already-aligned blobs as coloured circles onto the 2048²-resized `*_mhi.png`. Both resize the MHI background to 2048² so it matches the blob coordinate space. The archived `mhi_overlay_v1.py` is the same uncorrected-branch overlay as v2 but **without** that resize, so its overlays misalign unless the MHI PNG is already 2048² (superseded by v2).

### The `kinematics/` folder — stage 7, split by functionality

The old stage-7 scripts were one 500-line module-level loop duplicated across both branches (v3 and v4 differ **only** in folder names and a few report strings). `kinematics/` splits that into six files — prompt, config, two pure modules, two runnable stages — and collapses the branch duplication into one parameter. **This is the only part of the pipeline where scripts import each other, and the only stage that is interactive rather than edited**; the imports are plain siblings, so run the stages from inside `kinematics/` (its directory must be on `sys.path`).

- **`kin_prompt.py`** — all terminal interaction, kept out of the analysis code. `resolve_settings(param_specs=...)` returns `{'root': Path, 'branches': [...], 'choice': str, 'params': dict}` from, in precedence order: command-line arguments (`python extract_track_metrics.py <root> <branch>`), then an interactive prompt (root+branch pre-filled from the previous run), then a clean exit. The `params` dict holds the analysis parameters: each spec in `param_specs` (built by `kin_config.analysis_param_specs()` / `summary_filter_specs()`) is asked for **with its `kin_config.py` default pre-filled, never the last run's value** — Enter keeps the default, a typed value overrides it for this run only. Specs carry a `kind` (`float` / `int` / `int_or_none`, the last accepting `none` to disable) and optional `min`/`min_exclusive`, and bad input re-asks. Deliberate behaviours, each covered by a test: paths are quote/backslash/`~`-tolerant (VSCode "Copy as path" pastes quoted); a root without `segmentation_output/` is rejected and re-asked *before* the run starts; `both` expands to `["corrected", "uncorrected"]`; **parameters are prompted only on a genuinely interactive run — if both root and branch came from the command line the defaults are used with no prompting**, so the confirmation step and the parameter prompts both appear **only** if something was actually typed and fully-argument-driven runs never stall; `Ctrl-C`/EOF exits with a message, not a traceback; and when stdin is not a TTY and no arguments were given it exits with usage rather than hanging forever.
- **`.kin_last_run.json`** — written next to `kin_prompt.py`, holds the last root+branch so `Enter` accepts them next time. **Only root and branch are remembered — never the analysis parameters** (those always default from `kin_config.py`). **Gitignored** — it is the only place a machine-specific path is stored. Failure to write it is swallowed; it must never break an analysis.
- **`kin_config.py`** — not executed; the machine-independent constants the runnable stages import: the summary track filters (`MIN_TOTAL_TIME_S`, `MIN_DIRECTION_CHANGES`) and `SUMMARIZE_AFTER_EXTRACTION`. The **per-track analysis thresholds are *not* here** — they have a single home in `kin_metrics.DEFAULT_PARAMS`; `kin_config` only points the prompt at them. These constants are the **defaults** the prompts pre-fill and the values unattended runs use — edit them (the filters here, the thresholds in `kin_metrics.DEFAULT_PARAMS`) to change what is offered. `analysis_param_specs()` / `summary_filter_specs()` describe those parameters to `kin_prompt` (label/unit/kind/bounds + the default pulled from `DEFAULT_PARAMS` / the filter constants); 7a prompts both sets, standalone 7b prompts only the filters. `settings(branch, root_data_dir)` maps a branch name to its input/output folder names and report wording (`title_suffix`, `extra_note`) — the single source of the v3/v4 distinction. An unknown branch raises rather than silently defaulting. **Do not reintroduce `ROOT_DATA_DIR`/`BRANCH` here**; they were deliberately removed so no one's local paths land in git.
- **`kin_metrics.py`** (pure, no I/O) — `calculate_angle`, `smooth_data`, and `compute_track_metrics(x, y, t, params)`, which takes one track in µm/s and returns a metrics dict, or `{'skip_reason': ...}` for tracks that are too short / have no positive `dt` / lose all velocities to the speed filter. Analysis thresholds live in `DEFAULT_PARAMS` and are passed in, not read from globals — so a single `.npy` can be analysed from a REPL without touching the data tree.
- **`kin_grouping.py`** (pure, plus the report writer) — `extract_sample_group` (the `_sampleX_..._recY` naming convention), `calculate_group_averages` (mean / sample-std / SEM), `group_tracks` (builds the `{shortened_dir: {condition: [...]}}` structure plus the pooled `combined` key), and `write_summary_file`. `title_suffix`/`extra_note` reproduce the corrected branch's wording.
- **`extract_track_metrics.py`** (stage 7a, runnable — **the entry point**) — tree traversal, `sh_<N>_t<n>` track naming, CSV writing, progress/error reporting; then invokes 7b unless `SUMMARIZE_AFTER_EXTRACTION` is off. Its `__main__` loops over the chosen branches, so `both` produces both result sets in one invocation. `main()` itself takes explicit arguments and does no prompting, which is what keeps the equivalence tests able to drive it directly.
- **`summarize_track_metrics.py`** (stage 7b, runnable) — reads `processed_results[_2]/*_processed_results.csv` and writes the `_summary_statistics.txt`. Touches no `.npy`, so grouping rules, track filters (`MIN_TOTAL_TIME_S`, `MIN_DIRECTION_CHANGES` — no-ops by default) and report layout can be iterated in seconds; run it standalone for exactly that. Its prompt asks only for the two summary track filters, not the per-track analysis parameters, which are already baked into the CSVs.

Deliberate differences from v3/v4 (everything else is numerically identical — both branches' summary files were verified byte-for-byte against the old scripts on synthetic data):

- **CSV gained two columns** — `Shortened_Dir` and `Experimental_Condition`, so stage 7b can group without re-walking the tree. CSVs written by the old scripts are therefore rejected by `summarize_track_metrics.py` with a message telling you to re-run 7a.
- **CSV metric columns now carry 6 decimals instead of 3** — the summary now averages the CSV values rather than in-memory floats, and 3 decimals would round-trip into the report.
- **Encoding fixes.** The old scripts write `μ` to the summary with the platform default encoding, which **crashes on Windows** (cp1252); and their `SPEED FILTER` progress line also contains `μ`, so on a Windows console that `print` raised `UnicodeEncodeError` inside the `try`, and the bare `except` logged every speed-filtered track as a failed track. The new code opens report/CSV files with `encoding='utf-8'` and keeps console output ASCII.
- The unused `filtered_t` variable was dropped.

### `segmentation_checks/` — batch QA of segmentation output

Run on a segmentation-output tree (`shortened_*/…/t<number>.png`); each writes a problem report + `summary.txt` + `empty_outputs.txt`. Both detect empty (all-black) files identically but flag a **different** track defect, so both are kept:

- **`error_analysis_v1.py`** — flags **fragmentation**: any image with ≥2 disconnected white components (>10 px). Report: `unconnected_paths.txt`. Catches single tracks broken into pieces (the defect `segmentation_corrections` v2 *merges*).
- **`error_analysis_v3.py`** — flags **multiple distinct tracks**: DBSCAN-groups nearby components (`min_area=50`, `grouping_distance=250`) and flags only images with >1 group, so a fragmented single track passes. Report: `multiple_tracks.txt`. Adds `psutil`/`gc` memory management. Catches genuinely-fused/co-located tracks (the defect `segmentation_corrections` v1/v3 *split*). (The archived `_v2` is v3 without the memory management — identical detection.)

### `segmentation_corrections/` — manual per-image track fix-ups

These are **run by hand on a single hard-coded image path** (not part of the automated batch) to correct a track that stage-1 segmentation got wrong. They are **not a version progression** — each solves a *different* segmentation defect, so all three are kept:

- **`correction_v1.py`** (`separate_tracks_simple`) — **split** an image into its 8-connectivity connected components (each ≥ `min_area`), saved as `_1, _2, …`. Use when tracks are already physically separate. Its function is copied verbatim inside v2.
- **`correction_v2.py`** (`separate_tracks_with_grouping`) — the **opposite: merge**. DBSCAN-groups nearby disconnected components (`grouping_distance`) back into a single track. Use when one track was shattered into fragments. (Requires `sklearn`.)
- **`correction_v3.py`** (`separate_connected_tracks`) — **split the hard case: physically fused tracks** that connected-components can't separate. Runs a 4-method cascade in order until one yields >1 component: skeleton branch-points → erosion → contour direction-change → coordinate/junction. Output suffixed by method (`_skeleton_/_erosion_/_direction_/_coordinate_N`). (Requires `scipy` + `skimage`.)
- **`correction_v4.py`** (`merge_specific_files`, formerly `merge_v1.py`) — the **manual inverse of the split tools**: combines a **hand-listed** set of white-segment PNGs (e.g. `t12_1.png` + `t12_3.png`) into one image via pixel-wise `np.maximum`, saved as e.g. `t12_merged.png`. Unlike v2's automatic DBSCAN merge, you specify exactly which fragment files to combine (hard-coded list). Use to stitch chosen split-fragments back into a single track.

## Data-format invariants (critical when editing)

- **Blob NPY array**: shape `(n_blobs, 6)`, columns `[x, y, timepoint, area, brightness, solidity]`. Index 2 = timepoint is used everywhere for temporal logic. Preserve this column order — downstream stages index columns positionally.
- **Coordinate space**: blob detection resizes every image to **2048×2048** before detection, so all blob and MHI coordinates are in 2048×2048 space (aspect ratio of non-square originals is distorted). Track masks and MHIs are resized to match. Real-world conversion happens only in stage 6.
- **Per-subfolder blob parameters**: the blob detector holds a `subfolder_params` dict keyed by experiment name (`Lead_1`, `Copper_3`, …) that overrides `default_params` per experiment. Add new experiments there, not by forking the script.
- Scripts are written for large datasets and are deliberately **memory-conscious** — they `del` large arrays promptly and (in error analysis) call `gc.collect()` under `psutil` memory pressure. Keep this discipline when adding processing to the batch loops.
